## 設計邏輯（先讀懂再動手）
機構是一對多：一位督導管理多位個案，每位個案各自處於連動生命週期的某一態。
**同步給家屬端的服務紀錄，來源就是核銷流程中督導確認過的紀錄**（ocr_confirmed）
——不另造資料流。督導做核銷確認的那個動作，同時完成了家屬端資料的品質把關。
這是「一筆紀錄，三種價值」的產品機制化，UI 上要把這件事講出來。

連動生命週期：`未邀請 →(產生邀請碼)→ 邀請中 →(家屬輸入碼)→ 已連動 ⇄(解除)`

## 任務 2：個案連動管理頁
Mock 資料（寫死常數，正式版接 Supabase）：
```js
const CASES = [
  { caseId:"A141408XXX", name:"王奶奶", status:"connected",
    family:"王小姐（女兒）", inviteCode:"CARE01", lastSync:"6/27", syncedCount:4 },
  { caseId:"B222333XXX", name:"林爺爺", status:"invited",
    inviteCode:"CARE02", invitedAt:"6/25" },
  { caseId:"C111222XXX", name:"陳阿姨", status:"none" },
];
```
以卡片列或表格呈現，每列包含：

| 欄 | 內容 |
|---|---|
| 個案 | `王奶奶（A141408XXX）`（姓名＋代號） |
| 連動狀態 | badge 三態：`未邀請`(灰) / `邀請中 · CARE02`(琥珀，附複製鈕) / `已連動 · 王小姐（女兒）`(綠) |
| 服務紀錄同步 | 已連動：`最近同步 6/27 · 本月已同步 4 筆` + 按鈕〔推送本月紀錄〕；其他狀態顯示 `—` |
| 操作 | 未邀請：〔產生邀請碼〕→ 狀態變邀請中（碼由常數池取，如 CARE03）；邀請中：〔複製邀請碼〕〔重新產生〕；已連動：〔解除連動〕（confirm modal，確認後回未邀請態） |

互動行為（本階段皆為前端 state 模擬）：
- 〔推送本月紀錄〕→ toast：`已將 4 筆督導確認之服務紀錄同步至家屬端`
  （筆數取 syncedCount），並更新 lastSync 為今日。
- 〔產生邀請碼〕→ 開 modal 顯示大字邀請碼（沿用任務卡 B 的 modal 版型）
  + 複製鈕 + 引導文字。
- 所有 modal 禁用 window.confirm / alert。
