## 三、UI/UX 協作工作流（設計成員 → 開發模型）

**流程：tokens 表 → 標註截圖 → 樣式限定任務卡 → preview 驗收 → 合併**

### 3.1 Tokens 表（一頁，7/16 定稿，之後不再改）
設計成員填寫下表，開發側落地為 `globals.css` 的 `@theme` CSS 變數，
三端一律從變數取值：

```
主色（品牌）：#______（現行松綠 #1b4d48 為基準）
角色 accent：試算＝藍系 #______／家屬＝玫紅系 #______／機構＝松綠系 #______
中性色階：背景／卡片／邊框／主文字／次文字（5 個值）
語意色：success／warning／danger（3 個值）
字級階：h1 __px／h2 __px／正文 __px（≥16）／輔助 __px
圓角：卡片 __px／按鈕 __px｜陰影：1 組｜按鈕：主要／次要／危險 3 種樣式
```
規則：**三端底色、字階、圓角、陰影、按鈕統一；角色 accent 各端保留**
（對應流程圖 v3 的圖例色，簡報與產品一致）。

### 3.2 標註截圖規格
每張截圖標明：頁面路徑＋圈註區域＋期望改法（引用 tokens 名稱，
如「改用 主色」「字級升為 h2」）。一張截圖＝一個畫面，不跨頁。

### 3.3 樣式限定任務卡模板（複製給開發模型）
```
【UI 樣式卡 · <畫面名>】
1. 僅允許修改 className 與樣式相關 JSX 屬性；
   ❌ 禁止改動：任何邏輯、state、props、事件處理、文案、資料結構。
2. 依附件標註截圖執行；顏色/字級一律引用 globals.css 之 tokens 變數，禁止寫死色碼。
3. 驗收：(a) 對照截圖逐項確認 (b) 三端每個 tab/檢視各點擊一輪無 error
   (c) npm run build 通過 (d) 回報變更檔案清單。
```

### 3.4 驗收方式
開分支 → Vercel 自動產生 preview URL → 設計成員直接開網址對照標註圖驗收
→ 通過後 Jackson 合併 main。每日至多合併一次，合併後跑三端 smoke。

## 四、F1-Lite 任務卡（無金流・無真帳號・零 demo 風險）

```
【任務卡 F1-Lite】方案說明頁＋示意註冊
全域規則同 CLAUDE.md。修改檔案：src/app/plans/page.js（新增）、
src/components/FamilyDiaryV3.jsx、src/components/SubsidyCalculator.jsx（僅彈窗按鈕）。

任務 1：/plans 靜態方案頁（同時是商業模式簡報素材）
 兩欄卡片：
 【免費版】1 位被照顧者・日誌保存 90 天・AI 照護訊號・就醫摘要
 【進階版】多位被照顧者・日誌無限保存・PDF 匯出・家屬多人共編・AI 週報
 頂部標語：「先安心使用，需要更多再升級」；進階版按鈕〔即將推出〕(disabled)。
 頁尾小字：「方案內容以正式上線公告為準」。

任務 2：示意註冊（純前端 state，不接任何後端）
 /diary 首次進入（solo、未註冊 state）顯示引導卡：
 「建立帳號，開始記錄」＋email 輸入框＋〔開始使用〕→
 前端驗證 email 格式 → state 設 registered=true → toast「歡迎使用照護日誌」
 → 進入現有 solo 流程。header 顯示該 email 前綴。
 試算端日誌 CTA 彈窗底部加小字連結「查看方案說明」→ /plans。

驗收：/plans 渲染正確；示意註冊全程無網路請求（DevTools 確認）；
 註冊後日誌功能與現況完全一致；三端 smoke；npm run build 通過。
禁止：不得引入 Supabase Auth 或任何 auth 套件；不得阻擋已進行中的 demo 流程
 （dev 切換器直接以 registered=true 進入，排練不受影響）。
```
