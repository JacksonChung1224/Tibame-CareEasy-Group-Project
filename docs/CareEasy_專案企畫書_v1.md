# 照護一點通（CareEasy）專案企畫書
> 版本：v1.0｜2026-07-13｜Tibame 專題團隊
> 線上平台：care-easy.vercel.app

---

## 一、專案摘要

**照護一點通**是一個台灣長照 3.0 資訊整合平台，以「**一筆紀錄，三種價值**」為核心設計理念：居服員本來就必須填寫的一筆服務紀錄，在本平台上同時成為機構的核銷憑證、家屬的照護透明度來源、以及 AI 訊號分析的交叉驗證資料——不增加任何人的工作量，卻讓同一筆資料產生三次價值。

平台由三個端點組成，共用同一套資料與規則引擎：

| 端點 | 對象 | 核心功能 | 使用門檻 |
|---|---|---|---|
| 補助試算平台 | 一般大眾 | CMS 等級推估、四包錢補助試算、服務廠商導流 | 免登入 |
| 照護日誌（家屬端） | 家庭照顧者 | 每日快速紀錄、AI 照護訊號、就醫摘要、機構連動 | 帳號 |
| 機構管理台（機構端） | 居服機構督導 | 排班對照、OCR 輔助核銷、衛福部官方格式匯出、個案家屬連動管理 | 機構帳號 |

## 二、問題背景

**家庭照顧者端**：長輩出院返家後的 7–14 天是照顧安排的關鍵空窗期——家屬在焦慮中同時面對「能領多少補助、該找哪些服務、怎麼判斷長輩狀況有沒有變差」三個問題，而現有資訊分散在政府網站、電話專線與口耳相傳之間。日常照顧開始後，家屬對居服員到底做了什麼、長輩狀況是否緩慢惡化，多半只能靠打電話問機構或憑印象判斷，等到明顯惡化才就醫或申請重評，往往已錯過調整照顧計畫的時機。

**居服機構端**：機構每月核銷仍大量依賴居服員的紙本服務紀錄，督導需將紙本逐筆與排班系統人工比對後，繕打成衛福部規定格式申報。漏項、時數不符、計畫外服務未確認等問題耗費大量人力，比對疏漏更直接構成申報風險。

**制度端**：長照 3.0（民國 115 年上路）調整了給付架構與服務組合表，資訊更新快、規則複雜，一般家庭難以自行理解 CMS 等級與「四包錢」額度的關係。

## 三、解決方案

### 3.1 補助試算平台——資訊斷層的第一站
使用者依失智（CDR）／非失智（Barthel 加權）雙軌完成評估問卷，平台以規則引擎**推估** CMS 等級（2–8 級），並展開四包錢（照顧及專業服務、交通接送、輔具與居家無障礙、喘息服務）之每月**最高**補助額度與細項用途。所有金額均標示「實際額度須經各縣市長期照顧管理中心評估核定」，並引導使用者撥打 1966 進行正式評估。細項服務可直接導流至合作廠商，形成從「知道額度」到「取得服務」的完整動線。

### 3.2 照護日誌（家屬端）——把日常觀察變成可行動的訊號
家屬每天以快速標籤（嗆咳、睡不好、走路沒力等）加選填文字完成紀錄，十秒即可完成。累積的紀錄由**規則引擎**（非黑箱模型）分析：每一條照護訊號都對應明確規則與原始紀錄依據，家屬可逐筆檢視「AI 為什麼這樣說」。輸入機構邀請碼連動後，家屬可直接查看督導確認過的居服員服務紀錄與即時額度；AI 進一步交叉比對家屬與居服員兩端觀察，**兩端一致的退化訊號**以高可信度標示，並在達到門檻時引導家屬與照管專員討論是否申請提前複評——再由試算平台預估新等級，形成完整循環。

**隱私設計為單向不對稱**：服務紀錄由機構授權家屬檢視；家屬日誌永不提供機構，僅 AI 彙整之訊號可經家屬主動同意分享。這是家屬敢於誠實記錄的前提。

### 3.3 機構管理台（機構端）——從紙本到申報檔的最後一哩
督導上傳排班計畫（支援衛福部標準格式），系統以**兩輪配對演算法**（個案＋日期＋服務碼精確配對，再處理同日項目差異）自動比對紙本實績，差異以五種狀態分類：一致、疑似未執行（D1）、計畫外服務（D2）、數量差異（D3）、項目差異（D4）。D1 永不進入核銷明細、D2 須督導確認，貫徹「以紙本實績為唯一申報基準」。單價與衛福部附表不符時自動亮警示但不改值，把稽核判斷權留給督導。OCR 輔助辨識紙本後，所有低信心欄位強制人工逐欄確認才能匯入。最終一鍵匯出**衛福部官方核銷格式**（紅字必要 11 欄，表頭一字不差），異常明細另附稽核附表。「個案連動管理」頁讓督導以個案為單位管理家屬連動生命週期（邀請、連動、推送紀錄、解除），一對多場景完整支援。

### 3.4 三端飛輪
試算（獲客）→ 廠商導流（媒合）→ 機構開案 → 邀請碼連動 → 日誌＋居服紀錄雙軌 → AI 訊號 → 提前複評討論 → 回到試算——每一圈循環都讓平台上的資料與信任加深一層。

## 四、核心差異化

1. **一筆紀錄，三種價值**：不要求任何角色多做事，而是讓既有的法定紀錄流動起來。督導在核銷流程中確認紀錄的動作，同時完成了家屬端資料的品質把關。
2. **可解釋的規則引擎，不是黑箱 AI**：CMS 推估與照護訊號皆為明文規則，每個結論附原始依據；規則閾值由專責的法規查核角色以測試案例驗證，而非工程師自由心證。
3. **合規內建於產品**：金額一律標示「最高」與「須經評估核定」、AI 分析明示「觀察彙整、非醫療診斷」、費率變更須通過查核清單簽核才能上線、疑似未執行服務在架構上就無法進入申報檔。
4. **與政府申報格式直接相容**：匯出檔即為衛福部範本之必要欄位格式，機構無須二次繕打。

## 五、商業模式

採「先建立信任流量、再分層變現」的漏斗策略：

1. **免登入試算＝獲客入口**（不收費）：以出院轉銜期的高焦慮需求為切入點，建立第一次接觸。
2. **服務媒合導流**：試算結果的細項服務導流合作廠商，未來以轉介分潤或「出院轉銜照護組合」形式變現；平台已建立導流點擊追蹤，可量化媒合成效。
3. **照護日誌 Freemium**：基礎功能免費（單一個案、有限保存期），進階方案提供多個案、無限保存、就醫摘要 PDF、家屬多人共編與 AI 週報。
4. **B2B 機構訂閱（中長期主軸）**：核銷比對與官方格式匯出對機構是明確的人力節省，以訂閱制向居服機構收費；CMS 推估引擎並可白牌授權予照護機構作為評估輔助工具。

## 六、技術架構

- **前端／部署**：Next.js（App Router）＋ React ＋ Tailwind，Vercel 部署，三端合一 monorepo、單一網址。
- **資料**：Supabase（評估紀錄含「推估等級 vs 實際核定等級」真值配對，供引擎持續回測；導流點擊追蹤；寫入經 server route 隔離）。
- **規則引擎層**（皆為純函式、零依賴、可獨立測試）：CMS 等級推估、照護訊號引擎（R1–R4 規則）、OCR 後處理校正（服務碼白名單模糊校正＋生理值範圍驗證）、核銷兩輪配對引擎、官方格式匯入解析器。
- **品質體系**：四套 Node 直跑測試（現行 48+ 案例全綠）＋ 歷史案例回測腳本；「測試檔即驗收介面」讓非工程成員也能驗證 AI 產出的程式碼。
- **AI 協作開發**：以任務卡制度（自包含規格＋驗收條件＋禁改清單）驅動 AI 開發模型執行，配合 CLAUDE.md 專案規則自動載入，形成可複製的單人技術執行×多人驗收的開發模式。

## 七、合規與風險管理

| 風險 | 對策 |
|---|---|
| 費率／法規更新 | 費率驗證清單制度：所有金額與法規文案變更須經 CMS 法規查核角色對官方文件逐項簽核；CHANGELOG 專設費率追蹤區塊 |
| 過度承諾疑慮 | 全站用詞紀律：「推估」非「判定」、「最高」冠於金額、「須經評估核定」、AI 為「觀察彙整、非醫療診斷」 |
| 申報正確性 | D1 架構性排除於申報檔、單價稽核警示、低信心 OCR 欄位強制人工確認、以紙本實績為唯一基準 |
| 個資與隱私 | 家屬日誌單向隱私（機構不可見）、身分證號去識別化展示、分享須明示同意且可撤回 |
| 單人技術執行 | 純函式＋測試檔驗收機制、任務卡協作協議、規則與程式碼分離 |

## 九、團隊分工

六人團隊採「單一技術執行＋分工驗收」模式：技術與產品規格（PRD、引擎規格、系統流程、UAT）由開發負責人統籌，AI 開發模型依任務卡執行；法規查核角色以測試案例驗證所有補助規則產出；其餘成員分別負責使用者研究（人物誌、旅程圖）、專案管理（章程、甘特、RACI）、商業分析（商模畫布、競品）與風險溝通（利害關係人、風險登錄、簡報）。

## 十、未來規劃

短期（決賽後三個月）：OCR 視覺辨識正式化並以欄位級準確率持續量測；累積「推估 vs 核定」真值資料回測並校準推估引擎；日誌 Freemium 帳號體系與方案上線。中期：居服機構試點合作，驗證核銷模組的實際人力節省；家屬同意制的 AI 摘要分享，補齊機構端在連動關係中的回饋價值。長期：以「出院銜接層」定位深化與醫院出院準備服務的介接，並將 CMS 推估引擎白牌化為照護機構的評估輔助工具。
