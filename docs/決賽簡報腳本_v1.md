## 第 1 段｜開場・問題洞察（3:00）

**口白骨架**（六大標準第一條：一句話說清楚誰在什麼情境遇到什麼問題）：
> 「長輩出院後的第 7 到 14 天，是台灣 XX 萬個家庭照顧者最焦慮的兩週——
> 能領多少補助？該找什麼服務？長輩狀況是不是在變差？沒有人給他們一個入口。」

內容順序：
1. 一句話題目（上句）→ 放大到兩個角色的痛點：
   - 家屬：資訊分散、額度看不懂、日常只能靠印象與電話追問。
   - 機構督導：紙本核銷逐筆人工比對繕打，漏項即申報風險。
2. JTBD 與替代方案（引用團隊 UVP&JTBD 交付物）：現況替代方案是
   1966 專線＋親友口耳＋Excel／紙本——各解一角，沒有閉環。
3. 收束：「我們的答案不是再做一個 App，而是讓一筆本來就存在的紀錄流動起來。」
   （為第 2 段鋪 punchline）

**投影片素材**：旅程圖 v3 第一段（家屬 7 站含情緒線）截圖；痛點數據頁。

## 第 2 段｜解決方案・產品思維（2:30）

1. **核心架構一頁**：旅程圖 v3 第四段「一筆紀錄，三種價值」三叉圖。
   口白：「居服員本來就必須寫的一筆紀錄——在我們平台上同時是
   機構的核銷憑證、家屬的照護透明度、AI 的交叉驗證資料。零額外工作量，三次價值。」
2. **MVP 範圍與 Trade-off**（產品思維得分核心，講「不做什麼」）：
   - 不做黑箱 ML——用可解釋規則引擎，每個訊號附原始依據（信任＞炫技）。
   - 不做多邊市場——先做單一機構白牌路線（引用 MVP 範圍表）。
   - 砍掉「計畫外服務」分類——訪談確認實務不會發生（真實迭代證據）。
3. **驗證假設與成功指標**：試算→日誌轉換率、機構核銷工時節省、
   「推估 vs 實際核定等級」準確率（真值管線已在收資料）。

## 第 3 段｜Live Demo 六幕（7:00）— 操作 Jackson、口白一人

> 事前準備：dev 參數進入預設狀態；備援影片同步待命；網路備用熱點。

### 幕一（5:30–6:20）出院第一站：試算
- 操作：首頁三入口 → 試算 → 快速完成評估（預先想好答案組合）→ CMS 4 級。
- 口白：「免登入、三分鐘，家屬第一次看懂自己『最高』能有哪些資源。
  注意等級是鎖定的——推估結果不能隨意改，這是我們的合規設計。」
- 指出：四包錢細項＋合作廠商導流＋「所有金額標示最高、須經評估核定」。

### 幕二（6:20–7:00）日誌入口
- 操作：四包錢明細上方 📓 CTA → 說明彈窗 → 前往日誌 → 示意註冊（email）。
- 口白：「試算是入口，日誌是留下來的理由。免費開始，這是我們漏斗的第二層。」

### 幕三（7:00–8:20）機構端：督導的一天
- 操作：機構端 → 上傳**真實衛福部排班檔** → OCR 步驟：上傳紙本照片
  →「載入 OCR 模擬資料」（口徑：辨識引擎示意）→ 低信心欄標黃 → 逐欄確認
  → **手寫觀察區**：確認「案主吃早餐時不慎跌倒…」。
- 口白：「同一張紙本、兩條資料流：核銷欄位走申報，手寫觀察走照護。
  低信心欄位強制人工確認——AI 輔助，人負責。」

### 幕四（8:20–9:40）對帳裁決與一鍵申報
- 操作：比對室 → 指出五色狀態 → D3 列示範「採紙本／採系統」radio →
  D1 填未執行原因 → 匯出 → **開啟 xlsx**：官方 11 欄核銷明細＋異常附表。
- 口白：「兩輪配對演算法自動比對，督導只裁決差異。疑似未執行的服務，
  在架構上就進不了申報檔。匯出的就是衛福部格式，不用二次繕打。」

### 幕五（9:40–11:00）連動：一筆紀錄開始流動
- 操作：個案連動管理 → 對陳阿姨產生邀請碼 → 展開王奶奶「已確認服務紀錄
  ＋照護觀察」→ 推送 → 切家屬端 → 輸入邀請碼 → connected →
  居服紀錄卡出現**同一句跌倒觀察**。
- 口白：「督導剛剛確認的那句話，現在出現在女兒的手機上。
  她不用再打電話問『媽媽今天怎麼樣』。而她自己的日誌，機構永遠看不到——
  單向隱私，是家屬敢說真話的前提。」

### 幕六（11:00–12:30）AI 訊號與飛輪閉合
- 操作：先秀 solo 模式 AI 分析（兩個琥珀單邊訊號）→ 切 connected →
  同一份日誌變**兩個紅色「兩端一致」**（行走含跌倒交叉、吞嚥）→
  重評建議卡 → 點「用試算平台先估新等級」回到幕一入口。
- 口白：「同樣的日誌，連動前是兩個單邊猜測，連動後是兩端一致的高可信訊號
  ——這就是連動的價值。每個訊號都能展開原始依據，AI 不是黑箱。
  最後這顆按鈕把家屬帶回試算——飛輪閉合。」

## 第 4 段｜AI 協作與工具運用（2:30）— 15% 的獨門彈藥

講你們**怎麼做出來的**（評審少見的真實 AI 工程管理）：
1. **任務卡制度**：每張卡＝自包含規格＋資料結構＋驗收條件＋禁改清單，
   交給 AI 開發模型執行；秀一張實際任務卡截圖。
2. **CLAUDE.md**：專案規則自動載入，每個 AI 會話開場就守規矩。
3. **測試檔即驗收介面**：48+ 個 Node 直跑測試——**非工程成員跑一行指令
   就能驗收 AI 寫的程式碼**；秀一次測試全綠的終端機畫面。
4. **合規閘門**：涉及費率與法規文案的 AI 產出，一律經 CMS 法規查核
   角色對官方文件簽核才能上線（費率驗證清單截圖）。
5. 收束：「一人技術執行 × AI 模型代工 × 全隊可驗收——這套方法本身，
   就是 AI 時代的專案管理答案。」（呼應六大標準「AI 時代差異」）

## Q&A 備答（預測提問）

| 可能提問 | 回答要點 |
|---|---|
| 為什麼不用機器學習？ | 照護決策需可解釋與可稽核；規則引擎每訊號附依據、閾值由法規角色驗證；已建真值管線，資料足夠後可回測校準——先信任後智慧。 |
| OCR 準確率多少？ | 現為示意輸出；架構已就緒（後處理校正＋信心分數＋強制人工確認）；正式化後以「欄位級準確率」量測，且低信心欄永遠過人工。 |
| 個資怎麼處理？ | 蒐集前告知＋未預勾同意；家屬日誌單向隱私；身分證去識別化展示；分享須明示同意可撤回。 |
| 機構為什麼付錢？ | 核銷比對工時可量化（每月 X 小時人工比對→分鐘級）；匯出即官方格式免繕打；申報風險下降。 |
| 跟 1966／照專是什麼關係？ | 互補不取代：我們是入口與日常層，正式評估與核定永遠導向照管中心；所有文案標示須經評估核定。 |
| 法規改了怎麼辦？ | 費率驗證清單＋CHANGELOG 費率追蹤＋合規閘門；114/06/19 修正即為實戰案例。 |
| Jubo 是競品還是夥伴？ | 定位互補：他們深耕機構內系統，我們補「機構×家屬×政府申報」的縫隙；導流合作即證明。 |
| 資料真實性？ | 排班檔用衛福部真實範本實測；手寫觀察取材真實紙本紀錄表；評估紀錄已在累積真值。 |
